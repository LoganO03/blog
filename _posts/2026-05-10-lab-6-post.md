#### User Stories
User Stories use a Size of 2.5 hours per 1 size value.

1. As the store owner, I want to be able to list items for sale so that registered customers can buy them.
- Related to stories: 3, 4
- Size: 4

2. As an unregistered customer, I want to be able to register an account so that I can buy items.
- Related to stories: 3,4
- Size: 6

3. As a registered customer, I want to be able to start an order so that I can receive items.
- Related to stories: 4
- Size: 5

4. As a customer, I want to be able to add items to my order so that I can finalize my order.
- Related to stories: 3, 5
- Size: 8

5. As the owner, I want to be able to view finalized orders so that I can get them filled for completion.
- Related to stories: 4
- Size: 2

#### Reflection
##### User Stories
The user stories were not too difficult to come up with. Thinking as the customer and the owner/admin made it easy to figure out what kind of details each would need to have in order for the website/DB to be usable. I slected to mimic the Size attribute that Agile used for determining story work times. I feel the Size works well for each story as iventory feels relativly easy to implement in comparison to that of the regisration and order processes for customers.
##### Entity Relationship Diagram
When it comes to ERD, I feel this diagram works well and was easy to implement into the Schema. I decided to add ID values to each of the entities as if felt like a solid idea to keep the tables simple insted of using complex primary keys. For each entity I have assigned all the attributes described in the description to my knowledge. I feel the most difficult part of this process was the cardinality, but I feel my cardinalities are solid.
##### Database Schema
When implementing the schema from the ERD, I feel the ERD I had designed was easy to follow. Though relationship tables do feel complex to consider as they can sometimes feel slightly abstract. Overall, I am pretty satisfied with my model. Issues that could come up is handling inventory concrurrencey. Meaning that if the stock is low and two customers ask for the remaing stock, then handling that overalap might be difficult in terms of who gets the items. Limits can be difficult to implement as orders in bulk could be difficult, each item may need a max compacity. Order Type as a boolean could be confusing, but if worked out properly could be displayed right. Substitution seems vauge, might be worth expanding the item table to include alternate item IDs. I feel payment and security would be difficult to implemtn as well, but not necessarily a database problem.
